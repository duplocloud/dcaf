# System Events

DCAF automatically emits `intermittent_update` stream events for key moments in
agent execution — when the model starts reasoning and when a tool begins executing.
This page explains how to configure, customise, and extend those built-in signals.

---

## What Are System Events?

While your agent is running, DCAF emits transient status messages to the client UI
at certain points in the lifecycle. These are called **system events** — they are
generated by the framework itself, not by your tool code.

By default, two are active:

| When it fires | Default text |
|---|---|
| Model begins reasoning | `"Thinking..."` |
| A tool begins executing | `"Calling tool: {tool_name}"` |

These appear on the client as `intermittent_update` stream events — the same type
that [`emit_update()`](./emitting-events.md) sends from your code. The UI shows
them while the agent is working and clears them when the next piece of content
arrives.

---

## The `SystemEvent` Descriptor

Each system event is represented by a `SystemEvent` object. Import the pre-built
constants — these are the intended public API:

```python
from dcaf.core import THINKING, TOOL_STARTED, TOOL_COMPLETED, TOOL_FAILED
```

| Constant | Fires when | Default text | Available variables |
|---|---|---|---|
| `THINKING` | Model begins reasoning | `"Thinking..."` | — |
| `TOOL_STARTED` | A tool begins executing | `"Calling tool: {tool_name}"` | `{tool_name}` |
| `TOOL_COMPLETED` | A tool finishes successfully | `"Done: {tool_name}"` | `{tool_name}` |
| `TOOL_FAILED` | A tool raises an exception | `"Failed: {tool_name}"` | `{tool_name}`, `{error}` |

`THINKING` and `TOOL_STARTED` are **on by default**. `TOOL_COMPLETED` and
`TOOL_FAILED` are **off by default** — opt in when you want them.

---

## Configuring System Events

Pass `system_events` to the `Agent` constructor.

### Keep the defaults (no configuration needed)

```python
agent = Agent(tools=[...])
# Equivalent to:
agent = Agent(tools=[...], system_events=None)
```

### Disable all system events

```python
agent = Agent(tools=[...], system_events=False)
```

### Choose specific events

```python
from dcaf.core import THINKING, TOOL_STARTED, TOOL_COMPLETED

agent = Agent(
    tools=[...],
    system_events=[THINKING, TOOL_STARTED, TOOL_COMPLETED],
)
```

Only the events in the list will fire. The default two (`THINKING`, `TOOL_STARTED`)
are not added automatically when you pass a list — you are taking full control.

## Customising the Text

### Simple text override

Replace the default string with your own. Template variables in `{curly_braces}`
are still supported:

```python
from dcaf.core import THINKING, TOOL_STARTED, TOOL_COMPLETED

agent = Agent(
    tools=[...],
    system_events=[
        THINKING.with_text("Working..."),
        TOOL_STARTED.with_text("Running {tool_name}..."),
        TOOL_COMPLETED.with_text("{tool_name} complete"),
    ],
)
```

### Custom formatter function

For dynamic logic — branching, lookups, or anything a template can't express:

```python
from dcaf.core import TOOL_STARTED

agent = Agent(
    tools=[...],
    system_events=[
        TOOL_STARTED.with_formatter(lambda d: f"▶ {d['tool_name']}"),
    ],
)
```

The formatter receives a `dict` containing the available variables for that event
(see the table above) and must return a `str`.

---

## Internationalisation (i18n)

Because formatters accept any callable, you can plug in any translation system.
The `data` dict is the same shape regardless of language:

```python
from dcaf.core import THINKING, TOOL_STARTED, TOOL_COMPLETED, TOOL_FAILED

def _(key: str, **kwargs) -> str:
    """Your translation function."""
    return translations[locale][key].format(**kwargs)

agent = Agent(
    tools=[...],
    system_events=[
        THINKING.with_formatter(lambda _d: _("agent.thinking")),
        TOOL_STARTED.with_formatter(lambda d: _("agent.tool_started", tool_name=d["tool_name"])),
        TOOL_COMPLETED.with_formatter(lambda d: _("agent.tool_done", tool_name=d["tool_name"])),
        TOOL_FAILED.with_formatter(lambda d: _("agent.tool_failed",
                                                tool_name=d["tool_name"],
                                                error=d.get("error", ""))),
    ],
)
```

## Opting Into Optional Events

`TOOL_COMPLETED` and `TOOL_FAILED` are not in the defaults to avoid doubling the
noise for every tool call. Add them explicitly when you want the full lifecycle:

```python
from dcaf.core import THINKING, TOOL_STARTED, TOOL_COMPLETED, TOOL_FAILED

agent = Agent(
    tools=[...],
    system_events=[
        THINKING,
        TOOL_STARTED,
        TOOL_COMPLETED,
        TOOL_FAILED.with_text("Error in {tool_name}: {error}"),
    ],
)
```

With this configuration the client receives four potential update events per tool:

```
TOOL_STARTED fires  → "Calling tool: list_pods"
TOOL_COMPLETED fires → "Done: list_pods"

TOOL_STARTED fires  → "Calling tool: delete_pod"
TOOL_FAILED fires   → "Error in delete_pod: permission denied"
```

## Relationship to `emit_update()`

System events and user-defined events are complementary. System events fire
automatically from the framework; user-defined events let your tool code push
any message at any time:

```python
from dcaf.core import emit_update, tool

@tool(description="Search for pods")
def search_pods(query: str) -> str:
    # Your code pushing a status update (user-defined)
    emit_update(f"Searching for: {query}")
    results = _do_search(query)
    emit_update("Search complete", content={"count": len(results)})
    return format_results(results)

# The framework automatically adds:
#   "Calling tool: search_pods"   ← system event (TOOL_STARTED)
# before your tool runs
```

See [Emitting Events](./emitting-events.md) for the full `emit_update()` /
`emit()` reference.

## Quick Reference

| Goal | Code |
|---|---|
| Keep defaults | `Agent(tools=[...])` |
| Disable all | `Agent(..., system_events=False)` |
| Choose events | `Agent(..., system_events=[THINKING, TOOL_STARTED])` |
| Custom text | `TOOL_STARTED.with_text("Running {tool_name}...")` |
| Custom formatter | `TOOL_STARTED.with_formatter(lambda d: ...)` |
| Add tool completion | `system_events=[..., TOOL_COMPLETED]` |
| Full lifecycle | `system_events=[THINKING, TOOL_STARTED, TOOL_COMPLETED, TOOL_FAILED]` |
